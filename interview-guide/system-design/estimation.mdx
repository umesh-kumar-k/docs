Chapter 2 covers back-of-the-envelope (BOTE) estimation fundamentals for system design interviews, focusing on scalability basics like data units, latencies, availability, and practical examples like Twitter QPS/storage.[1]

## Key Topics Section-Wise
- **Power of Two**: Binary data units (bit → byte → KB → MB → GB → TB → PB → EB → ZB) for accurate volume calculations; ASCII char = 1 byte (8 bits).[1]
- **Latency Numbers**: Jeff Dean's 2010 baselines (L1 cache 0.5ns, RAM 100ns, disk seek 10ms, etc.) with 2020 visualization; key insights: memory fast/disk slow, compress before network.[1]
- **Availability Numbers**: "Nines" (99.9% = ~9hrs/year downtime); SLAs from AWS/Google/Azure at 99.9%+.[1]
- **Example: Twitter Estimation**: 300M MAU → 150M DAU → 3500 QPS (peak 7000); media storage 55PB over 5 years.[1]
- **Tips**: Round/approximate, write assumptions/units, practice common calcs (QPS, storage, servers).[1]

## Tradeoffs
| Aspect | Pros | Cons/Limitations |
|--------|------|-----------------|
| **Precision vs Speed** [1] | Rounding simplifies interview math (100K/10=10K) | Over-approximation risks unrealistic designs |
| **BOTE vs Exact** [1] | Quick feasibility check; focuses process over result | Outdated latencies (e.g., Dean's 2010 numbers faster now) |
| **High Availability** [1] | More nines = better uptime (99.99%=~5min/year) | Exponential cost/complexity (5 nines needs heavy redundancy) |

## Design Patterns/Best Practices
- **Estimation Process**: State assumptions → calculate step-by-step → label units → round aggressively.
- **QPS Formula**: DAU × actions/day ÷ (24×3600) × peak factor (e.g., 2x).
- **Storage**: Daily volume × retention × growth factor.
- **Avoid Disk Seeks**: Favor memory/cache; compress for network.
- **Write Assumptions**: Clarifies reasoning; units prevent errors (KB vs MB).

## Interview Cheat Sheet (Q&A)
- **Q: What's BOTE?** A: Quick estimates using assumptions + common numbers to validate designs (process > precision).[1]
- **Q: Key latencies?** A: L1 0.5ns, RAM 100ns, SSD 150μs, disk seek 10ms, pack A→B 200ms.[1]
- **Q: Twitter QPS?** A: 300M MAU ×50% DAU ×2 tweets/day ÷86.4K sec ×2 peak =7K QPS.[1]
- **Q: 1PB storage?** A: 1024TB = 1M GB; daily 30TB media ×365×5yrs=55PB.[1]
- **Q: 99.9% vs 99.99%?** A: 9hrs vs 52min annual downtime; more nines = redundancy cost.[1]
- **Q: Estimation tips?** A: Round (100K/10), units (MB), assumptions first, peak factor 2x.[1]
