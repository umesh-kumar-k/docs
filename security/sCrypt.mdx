**scrypt** is a **memory-hard key derivation function (KDF)** designed to resist ASIC/GPU brute-force attacks by requiring massive memory (N parameter) during hashing, using **PBKDF2-style iterations** with **custom Salsa20/8 mixing**—parameters `N=2^14, r=8, p=1` (moderate) to `N=2^20` (high-security) trade compute time/memory for ASIC resistance vs fast SHA256.[medium] Senior architects use scrypt (N=2^17) for high-value wallets/server-side passwords, preferring Argon2id (modern standard) for memory-hard + time-hard balance.

## Interview Summary
**Flow**: `scrypt(password, salt, N, r, p, dkLen)` → massive memory array → Salsa20/8 mixing (r rounds) → PBKDF2-style iterations (p passes) → derived key; **Memory-hard**: N=2^20 requires ~1GB RAM (ASIC-prohibitive); **Parameters**: N (memory), r (block size), p (iterations), dkLen (output).[medium] **Java**: BouncyCastle `SCrypt.generate()`.

## Keywords & Patterns
- **Parameters**: `N=2^14-2^20` (memory), `r=8` (mixing), `p=1-8` (iterations), `dkLen=32` bytes.[medium]
- **Core**: Salsa20/8 (stream cipher mixing), ROMix (memory-hard loop).[medium]
- **Format**: Derived key (binary), not self-contained like bCrypt.[medium]

## Common Trade-offs + Questions
scrypt: ASIC-resistant/memory-hard vs high RAM usage (DoS risk); N tuning: Secure vs UX (login time); vs bCrypt: Memory-hard vs compute-hard; vs Argon2: Proven vs modern (side-channel resistant).[medium]

**Example questions**:
- scrypt parameters? (`N=memory, r=mixing, p=iterations`).[medium]
- Memory-hard benefit? (ASICs can't parallelize GB-scale RAM).[medium]
- DoS risk? (High N → attacker forces GB allocation).[medium]
