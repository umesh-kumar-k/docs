Asynchronous systems decouple services via message queues (RabbitMQ/SQS) and event streams (Kafka/PubSub), enabling scalability, fault tolerance, and burst handling through pub-sub patterns, eventual consistency, and backpressure management.[designgurus] Senior architects justify async over sync for high-write workloads (e.g., notifications, analytics), trading debug complexity for resilience via dead-letter queues (DLQ), idempotency, and consumer lag monitoring.[1][2]

## Interview Summary
Core patterns: **Producer publishes events** → **Broker queues/partitions** → **Consumers process asynchronously** (multiple instances). **Tools**: Kafka (durable logs, partitioning), RabbitMQ (queues/exchanges), SQS (managed). **Challenges**: Ordering (partition keys), duplicates (idempotency/message IDs), backpressure (consumer scaling/DLQ), eventual consistency (outbox pattern).[designgurus][3]

## Keywords & Patterns
- **Messaging**: Pub-Sub (Kafka topics), Queue (RabbitMQ/SQS), Fan-out (exchanges).[1]
- **Durability**: At-least-once (acks), Exactly-once (idempotency/transactions).[designgurus]
- **Backpressure**: DLQ (failed messages), consumer lag (partition rebalance).[2]
- **Ordering**: Partition keys (user_id), FIFO queues (SQS).[3]
- **Patterns**: Outbox (DB + queue tx), Saga (distributed tx), CQRS/ES (event sourcing).[4]
- **Metrics**: Queue depth, consumer lag, retry rate, DLQ size.[designgurus]

## Common Trade-offs + Questions
Async: scales/bursty vs sync simple/debuggable; Kafka durable/high-throughput vs RabbitMQ flexible/routing; eventual consistency vs strong (outbox complexity); at-least-once (dedup needed) vs exactly-once (performance hit).[5][1]

**Example questions**:
- Notifications at 1M/sec? (Kafka partitioned topics + fan-out).[designgurus]
- Order events lost? (Outbox pattern + idempotency).[2]
- Consumer lag? (Horizontal scale + partition rebalance).[3]

| Pattern | Scalability | Durability | Ordering | Complexity |
|---------|-------------|------------|----------|------------|
| Kafka Pub-Sub | Excellent | Logs | Partition | High [3] |
| RabbitMQ Queues | Good | ACKs | FIFO | Medium [5] |
| SQS | Managed | 7 days | FIFO opt | Low [designgurus] |

## Use Cases
- **High Write**: User events → analytics (Netflix Kafka).[6]
- **Bursty**: Notifications, image processing (Uber SQS).[7]
- **Decoupled**: Order placed → email/ship/inventory (RabbitMQ).[designgurus]
- **Event Sourcing**: Audit trails, state reconstruction.[4]

## Big Tech Blogs & Efficiency Wins
Netflix TechBlog: Kafka Streams cut analytics batch jobs 90%, real-time dashboards via event sourcing. Uber Engineering: Async outbox eliminated sync DB locks during Black Friday (10x throughput). ByteByteGo: RabbitMQ → Kafka migration handled 300M msg/day with 99.99% durability.[7][6][3]

## Cheat-Sheet Q&A
**Q: Kafka ordering?** A: Partition key = same consumer (user_id).[3]
**Q: Duplicate handling?** A: Message ID + idempotency key (Redis SETNX).[designgurus]  
**Q: Consumer lag?** A: Add instances → auto-rebalance partitions.[1]
**Q: DLQ purpose?** A: Failed retries → manual inspection/replay.[designgurus]  
**Q: Outbox pattern?** A: DB tx + queue insert (2PC or polling).[2]
**Q: Exactly-once?** A: Idempotent consumers + dedup store.[3]

## Tools/Frameworks
- **Brokers**: Kafka (streams), RabbitMQ (AMQP), SQS/SNS (AWS), PubSub (GCP).[3]
- **Java**: Spring Kafka/Rabbit, Reactor (reactive).[8]
- **Consumers**: Kafka Streams, Flink, Spark Streaming.[6]
- **DLQ**: Separate topics/queues, alerting (Prometheus).[designgurus]
- **Orchestration**: Kubernetes HPA (consumer autoscaling).[1]

## Data Structures & Algorithms
**Partitioning**: Consistent hashing (key → partition). **Consumer Groups**: Offset management (committed positions). **Dedup**: Redis Hash `msg:<id>:processed`. **Lag Detection**: High-water mark - committed offset. **Saga**: State machine (FSM) + compensating tx. **CRDT**: Conflict-free merges (eventual consistency).[2][3]

## Key Highlights & Tradeoffs
- **Scale**: Horizontal consumers > sync bottlenecks.[designgurus]
- **Resilience**: DLQ/retries > cascading failures.[1]
- **Debug**: Tracing (Jaeger) + event logs > distributed calls.[2]
- **Consistency**: Eventual cheap vs strong expensive.[4]
- **Interviews**: "Kafka partitioned + outbox + DLQ + consumer groups".[3]
