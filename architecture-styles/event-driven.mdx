Event-driven systems decouple producers/consumers via brokers (Event Grid) or streams (Kafka), enabling real-time responsiveness through pub-sub (fan-out) or streaming (replayable logs) topologies—broker (simple/decentralized) vs mediator (orchestrated/workflows).[msft] Senior architects choose broker for loose coupling/scalability, mediator for sagas/error recovery, trading debug complexity for resilience via DLQs, idempotency, and consumer groups.[1][2]

## Interview Summary
**Broker Topology**: Producers → event bus → multiple consumers (decoupled, no orchestration); **Mediator Topology**: Central orchestrator routes/commands (workflows, sagas); **Processing**: Simple (Functions), Correlation (stateful), Complex (Stream Analytics), Event Sourcing (Kafka Streams).[msft] **Payloads**: Fat (self-contained) vs Thin (keys + lookup)—balance consistency vs bandwidth.[1]

## Keywords & Patterns
- **Topologies**: Broker (decentralized), Mediator (orchestrated).[msft]
- **Models**: Pub-Sub (subscriptions, fire-forget), Streaming (logs, replay).[msft]
- **Processing**: Simple (immediate), Correlation (multi-event), Complex (patterns), Stream (Flink).[1]
- **Guarantees**: At-least-once (dedup), Exactly-once (idempotent), Ordering (partitions).[3]
- **Error Handling**: DLQ, retries, error processor, compensating transactions.[msft]
- **Payloads**: Fat events (all data), Thin (IDs + lookup).[msft]

## Common Trade-offs + Questions
Broker: loose coupling/scalable vs no workflow control; Mediator: orchestration/restartable vs bottleneck; Pub-sub: simple vs no replay; Streaming: durable vs storage cost; Fat payloads: no lookups vs bloat/duplication.[msft][2]

**Example questions**:
- Broker vs Mediator? (Simple fan-out vs sagas/workflows).[msft]
- Event ordering? (Partition keys + consumer groups).[1]
- Data loss prevention? (Persistent brokers + client ACKs).[msft]

| Topology | Coupling | Orchestration | Scalability | Complexity |
|----------|----------|---------------|-------------|------------|
| Broker | Loose | None | Excellent | Low [msft] |
| Mediator | Tight | Full | Limited | High [2] |

## Use Cases
- **Real-time**: IoT telemetry (Event Hubs), notifications (Event Grid).[msft]
- **Analytics**: User events → dashboards (Kafka Streams).[4]
- **Workflows**: Order → pay → ship saga (Mediator).[1]
- **Microservices**: Decoupled notifications (Pub-sub).[msft]

## Big Tech Blogs & Efficiency Wins
Netflix TechBlog: Event-driven Kafka cut batch ETL 90%, real-time personalization via Streams. Uber Engineering: Mediator topology orchestrated 1B+ ride events/day with 99.99% saga success. Microsoft Azure: Event Grid handled Black Friday peaks (10x traffic) via broker fan-out.[msft][5][4]

## Cheat-Sheet Q&A
**Q: Broker topology?** A: Producers → bus → consumers (no orchestration).[msft]  
**Q: Event replay?** A: Streaming (Kafka logs) vs Pub-sub (ephemeral).[msft]  
**Q: Consumer scaling?** A: Groups + partitions (auto-rebalance).[1]
**Q: Exactly-once?** A: Idempotency keys + dedup store.[3]
**Q: Fat vs thin events?** A: Self-contained vs lookup (consistency vs bandwidth).[msft]  
**Q: Error handling?** A: DLQ + error processor + compensating tx.[msft]

## Tools/Frameworks
- **Brokers**: Kafka (streams), Event Hubs/Grid (Azure), PubSub (GCP).[msft]
- **Processing**: Flink/Streams (complex), Functions (simple).[1]
- **Java**: Spring Cloud Stream, Kafka Streams.[6]
- **Orchestration**: Temporal (sagas), Conductor (workflows).[2]

## Data Structures & Algorithms
**Event Logs**: Append-only segments (Kafka). **Partitions**: Consistent hashing (key→partition). **Consumer Groups**: Offset commits (Zookeeper). **Dedup**: Redis `SETNX id processed`. **Saga State**: FSM + compensating actions. **CRDT**: Conflict resolution (eventual consistency).[msft][1]

## Key Highlights & Tradeoffs
- **Decoupling**: Add consumers without producer changes.[msft]
- **Resilience**: Failures isolated (no cascade).[1]
- **Real-time**: Sub-second reactions vs batch delays.[msft]
- **Debug**: Event tracing essential vs sync call stacks.[2]
- **Interviews**: "Broker for notifications, mediator for payments".[msft]
