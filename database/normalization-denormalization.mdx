## Interview Summary
Database normalization organizes data to eliminate redundancy and anomalies via normal forms (1NF-5NF, target 3NF), while denormalization intentionally adds redundancy for query performance in read-heavy systems. Senior architects balance both: normalize for write-heavy OLTP/integrity, denormalize selectively for OLAP/scale, justifying via access patterns, storage costs, and consistency needs.[1][2][3]

## Keywords & Patterns
- **Normal Forms**: 1NF (atomic values, no repeats), 2NF (full key dependency), 3NF (no transitive deps), BCNF (determinant=candidate key), 4NF/5NF (multi/join deps).[4]
- **Anomalies**: Insert (can't add without parent), Update (partial changes inconsistent), Delete (lose unrelated data).[5]
- **Normalization**: PK/FK relationships, Joins for reads, ACID writes efficient.[3]
- **Denormalization**: Embed duplicates, Materialized views, Star/Snowflake schemas.[6]
- **Hybrid**: Normalized OLTP → ETL to denormalized OLAP.[6]

## Common Trade-offs + Example Questions
**Trade-offs**:
- Normalization: Integrity/storage savings vs. slow joins/heavy reads.[3]
- Denormalization: Fast reads/simple queries vs. redundancy/update anomalies/storage bloat.[2]
- 3NF ideal but over-normalize → complex schemas; under → perf issues.[6]

**Example Questions**:
- Normalize customer-orders to 3NF; when denormalize for analytics?[7]
- Trade-offs for social feed: embed likes vs. join?[5]
- Anomalies in unnormalized e-commerce schema?[5]

## Use Cases
- **OLTP (Banking)**: Normalize to 3NF for txns/integrity.[3]
- **Analytics/Reporting**: Denormalize star schema for fast aggregates.[6]
- **Real-time Feeds**: Denorm user data into posts for sub-sec reads.[5]
- **Data Lakes**: Normalize raw → denorm warehouses (Snowflake).[6]

## Big Tech Blog References
- Netflix: Normalized user tables → denormalized recs/analytics for query speed.[5][6]
- Uber: Normalized trips → denorm dashboards; 100x query speedup at 3x storage.[6]
- Amazon: Normalized inventory → denorm catalogs for e-comm scale.[6]

## Cheat-sheet, Q&A Style
- **Q: 1NF rule?** A: Atomic values, unique rows (no arrays).[4]
- **Q: 3NF vs BCNF?** A: 3NF no transitive; BCNF every determinant candidate key.[4]
- **Q: Denorm benefit?** A: Fewer joins, faster reads (reads >> writes).[3]
- **Q: Fix insert anomaly?** A: Normalize to separate entities.[5]
- **Q: Hybrid when?** A: OLTP normalized + OLAP denorm via ETL.[6]

## Data Structures & Algorithms
- **DS**: Tables/Relations (sets), Trees (joins/indexes), Graphs (relationships).[8]
- **Algos**: Functional dependency analysis, Join optimization (hash/nested-loop), Aggregation (denorm scans).[9]

**Highlights**: Target 3NF baseline; denorm hotspots (reads>80%); manage anomalies via app logic/ETL; measure perf impact.[3][6]
**Questions**: Design normalized OLTP → denorm OLAP pipeline? When violate 3NF? Impact on sharding?[1]
